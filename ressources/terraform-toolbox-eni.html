🔧

terraform init

Initialisation
terraform init # Mise à jour des providers terraform init -upgrade # Reconfiguration du backend terraform init -reconfigure # Backend distant depuis fichier terraform init -backend-config="prod.tfbackend"
📌 : Télécharge les providers dans .terraform/. Ce dossier est à exclure du git.
🔍

terraform plan

Plan d'exécution
terraform plan # Sauvegarder le plan terraform plan -out=myplan.tfplan # Cibler une ressource terraform plan -target=azurerm_resource_group.rg # Avec fichier de variables terraform plan -var-file="prod.tfvars" # Refresh-only (pas de modification) terraform plan -refresh-only
💡 : Symboles dans le plan : + création, - destruction, ~ mise à jour, -/+ remplacement.
🚀

terraform apply

Application
terraform apply # Sans confirmation interactive (CI/CD) terraform apply -auto-approve # Appliquer un plan sauvegardé terraform apply myplan.tfplan # Forcer la recréation d'une ressource terraform apply -replace=azurerm_linux_vm.vm # Avec fichier de variables terraform apply -var-file="prod.tfvars"
⚠️ -auto-approve : À utiliser uniquement en CI/CD ou tests. En prod, toujours valider le plan avant.
🗑️

terraform destroy

Destruction
terraform destroy # Sans confirmation terraform destroy -auto-approve # Cibler une ressource terraform destroy -target=azurerm_resource_group.rg
🚨 Attention : Supprime TOUTES les ressources gérées par Terraform. Irréversible sans backup.
🛠️

terraform validate & fmt

Qualité du code
# Vérification syntaxique terraform validate # Formatage du code (cohérence) terraform fmt # Formatage récursif terraform fmt -recursive # Voir les changements de formatage terraform fmt -diff # Vérifier sans modifier (CI) terraform fmt -check
✅ Best practice : Activer Format On Save dans VSCode avec l'extension HashiCorp Terraform.
🔎

terraform output & show

Consultation
# Voir tous les outputs terraform output # Voir un output spécifique terraform output vm_ip_address # Format JSON (scripts) terraform output -json # Voir l'état complet terraform show # Voir une ressource spécifique terraform show -json | jq

🧱 Blocs HCL

Référence des blocs parents Terraform avec exemples annotés

⚙️ Bloc terraform {} — Configuration du projet ▼
terraform { # Version de Terraform requise required_version = "~> 1.5" # Providers nécessaires (avec version fixée) required_providers { azurerm = { source = "hashicorp/azurerm" version = "~> 3.90" } random = { source = "hashicorp/random" version = "~> 3.6" } } # Backend distant (Azure Blob) backend "azurerm" { resource_group_name = "rg-tfstate" storage_account_name = "sttfstateprod" container_name = "tfstate" key = "prod/terraform.tfstate" } }
📌 : Ce bloc ne peut utiliser ni variables, ni fonctions, ni références. Uniquement des constantes.
🔌 Bloc provider {} — Configuration du fournisseur ▼
provider "azurerm" { features {} # Obligatoire pour azurerm subscription_id = var.subscription_id # tenant_id, client_id, client_secret si SP } # Provider avec alias (multi-région/subscription) provider "azurerm" { alias = "secondary" features {} subscription_id = var.secondary_sub_id } # Utilisation de l'alias dans une ressource resource "azurerm_resource_group" "secondary" { provider = azurerm.secondary name = "rg-secondary" location = "northeurope" }
🔐 Auth Azure : En local, utilisez az login. En CI/CD, définissez ARM_CLIENT_ID, ARM_CLIENT_SECRET, ARM_TENANT_ID, ARM_SUBSCRIPTION_ID en variables d'env.
🏗️ Bloc resource {} — Ressources d'infrastructure ▼
# Syntaxe: resource "TYPE" "NOM_TERRAFORM" {} resource "azurerm_resource_group" "rg" { name = "rg-${var.env}-demo" location = var.location tags = local.common_tags } # Meta-arguments disponibles sur toute ressource resource "azurerm_virtual_network" "vnet" { name = "vnet-demo" location = azurerm_resource_group.rg.location # dépendance implicite resource_group_name = azurerm_resource_group.rg.name address_space = ["10.0.0.0/16"] # Dépendance explicite depends_on = [azurerm_resource_group.rg] # Comportement du cycle de vie lifecycle { prevent_destroy = true # empêche le destroy ignore_changes = [tags] # ignore les changements de tags create_before_destroy = true # crée avant de détruire } }

Référence d'un attribut : TYPE.NOM.ATTRIBUT → ex: azurerm_resource_group.rg.id

🔎 Bloc data {} — Lecture de ressources existantes ▼
# Récupérer un RG existant (non géré par ce projet) data "azurerm_resource_group" "existing" { name = "rg-shared-infra" } # Récupérer l'ID d'une subscription Azure data "azurerm_subscription" "current" {} # Utiliser la data dans une ressource resource "azurerm_virtual_machine" "vm" { resource_group_name = data.azurerm_resource_group.existing.name location = data.azurerm_resource_group.existing.location }
💡 : Référence : data.TYPE.NOM.ATTRIBUT. Lecture seule, ne crée rien, ne modifie rien.
📥 Bloc variable {} — Variables d'entrée ▼
variable "env" { type = string description = "Environnement de déploiement" default = "dev" # Contrainte de validation validation { condition = contains(["dev", "staging", "prod"], var.env) error_message = "env doit être dev, staging ou prod." } } variable "db_password" { type = string sensitive = true # masqué dans les logs CLI } variable "servers" { type = map(object({ size = string zone = number })) } # Utilisation var.env | var.db_password | var.servers["web"].size
📤 Bloc output {} — Sorties ▼
output "vm_public_ip" { description = "Adresse IP publique de la VM" value = azurerm_public_ip.pip.ip_address } output "connection_string" { value = azurerm_mssql_server.sql.fully_qualified_domain_name sensitive = true } # Récupérer depuis CLI terraform output vm_public_ip terraform output -json
📌 : Les outputs ne sont disponibles qu'après un apply, jamais lors d'un plan.
🏠 Bloc locals {} — Variables locales ▼
locals { # Convention de nommage centralisée prefix = "${var.project}-${var.env}" # Tags communs appliqués partout common_tags = { Environment = var.env Project = var.project ManagedBy = "Terraform" UpdatedAt = formatdate("YYYY-MM-DD", timestamp()) } # Calcul dynamique vm_names = { for k, v in var.servers : k => "vm-${local.prefix}-${k}" } } # Utilisation name = "${local.prefix}-vnet" tags = local.common_tags
⚠️ : Recalculées à chaque plan/apply. Ne pas abuser : trop de locals rend le code difficile à lire.

🚀 Utilisation Avancée

count, for_each, modules, conditions — la puissance de Terraform

🔢

Meta-argument : count

# Créer 3 instances identiques resource "azurerm_resource_group" "rg" { count = 3 name = "rg-app-${count.index}" location = "westeurope" } # Conditionnel (0 ou 1 ressource) resource "azurerm_public_ip" "pip" { count = var.create_public_ip ? 1 : 0 name = "pip-demo" resource_group_name = azurerm_resource_group.rg[0].name location = "westeurope" allocation_method = "Static" } # Référencer la ressource azurerm_resource_group.rg[0].name # index 0 azurerm_resource_group.rg[*].name # tous (liste)
⚠️ : count utilise des index numériques. Supprimer un élément en milieu de liste = destruction/recréation des suivants. Préférer for_each.
🔁

Meta-argument : for_each

# for_each avec une map d'objects variable "subnets" { default = { web = { prefix = "10.0.1.0/24" } api = { prefix = "10.0.2.0/24" } data = { prefix = "10.0.3.0/24" } } } resource "azurerm_subnet" "subnets" { for_each = var.subnets name = "snet-${each.key}" # "web", "api", "data" resource_group_name = azurerm_resource_group.rg.name virtual_network_name = azurerm_virtual_network.vnet.name address_prefixes = [each.value.prefix] } # Référencer avec la clé azurerm_subnet.subnets["web"].id
✅ Préférer for_each : Utilise des clés stables → pas de destruction en cascade lors de modifications de la liste.

Modules — Création

# Structure d'un module modules/ network/ main.tf # Ressources variables.tf # Inputs outputs.tf # Outputs exportés README.md # Documentation # modules/network/variables.tf variable "prefix" { type = string } variable "address_space" { type = list(string) } # modules/network/outputs.tf output "vnet_id" { value = azurerm_virtual_network.vnet.id } output "subnet_ids" { value = { for k, s in azurerm_subnet.subnets : k => s.id } }

Renommage — bloc moved

# Renommer une ressource dans Terraform # sans détruire/recréer la ressource réelle moved { from = azurerm_resource_group.old_name to = azurerm_resource_group.new_name } # Ou via CLI (moins recommandé) terraform state mv \ azurerm_resource_group.old_name \ azurerm_resource_group.new_name
⚠️ : Sans moved, Terraform voit l'ancien nom comme "à détruire" et le nouveau comme "à créer" — ce qui supprimerait votre infra.
💾 Gestion du State

L'élément central de Terraform — commandes, backends, bonnes pratiques

ℹ️ Le state c'est quoi ? Un fichier JSON (terraform.tfstate) qui maintient la correspondance entre votre code HCL et les ressources réelles chez le provider. Terraform compare à chaque plan/apply l'état souhaité (code) vs l'état connu (state) vs l'état réel (provider).
📋

Commandes state essentielles

# Lister les ressources terraform state list # Détail d'une ressource terraform state show azurerm_resource_group.rg # Supprimer du state (pas de l'infra !) terraform state rm azurerm_resource_group.rg # Renommer/déplacer dans le state terraform state mv azurerm_resource_group.old azurerm_resource_group.new # Backup du state (backend distant) terraform state pull > backup_$(date +%Y%m%d).tfstate # Restauration terraform state push backup.tfstate # Synchroniser le state avec la réalité terraform apply -refresh-only
☁️

Backend distant — Azure

# Dans terraform.tf terraform { backend "azurerm" { resource_group_name = "rg-tfstate" storage_account_name = "sttfstate12345" container_name = "tfstate" key = "prod/myapp.tfstate" } } # Ou depuis un fichier (recommandé par env) # prod.tfbackend : resource_group_name = "rg-tfstate-prod" storage_account_name = "sttfstateprod" container_name = "tfstate" key = "myapp.tfstate" # Initialiser avec fichier backend terraform init -backend-config="prod.tfbackend"
✅ : Activer le versioning sur le container Azure Blob pour garder un historique des states.
🔒

Verrouillage du state

Le backend Azure utilise un bail sur le blob pour verrouiller le state lors d'un apply. Empêche deux exécutions simultanées.

# En cas de crash : forcer le déverrouillage terraform force-unlock LOCK_ID # L'ID du lock apparaît dans le message d'erreur # Ou le libérer depuis le portail Azure : # Storage Account > Container > Blob > Casser le bail
⚠️ : Ne jamais forcer le déverouillage si un apply est vraiment en cours. Risque de corruption du state.
🚨

Recréation forcée

# Forcer la recréation d'une ressource terraform apply -replace=azurerm_linux_virtual_machine.vm # Exemple : rotation d'un mot de passe random terraform apply -replace=random_password.db_pass # Ancienne méthode (taint) — dépréciée terraform taint azurerm_linux_virtual_machine.vm terraform untaint azurerm_linux_virtual_machine.vm
📌 : -replace est disponible lors d'un plan ET d'un apply. Toujours vérifier le plan d'abord car d'autres ressources peuvent être impactées.
🚨 Règles d'or sur le state
1. Ne JAMAIS commiter le state dans Git (*.tfstate dans .gitignore)
2. Ne JAMAIS modifier le state manuellement — utiliser les commandes terraform state *
3. Utiliser un backend distant dès qu'on travaille en équipe
4. Sécuriser les droits d'accès au backend (le state peut contenir des secrets en clair)

Organisation, sécurité, conventions — écrire du Terraform professionnel

📁

Organisation des fichiers

# Structure recommandée projet/ ├── terraform.tf # bloc terraform {} uniquement ├── providers.tf # tous les blocs provider {} ├── variables.tf # déclarations des variables ├── locals.tf # locales partagées ├── outputs.tf # toutes les sorties ├── main.tf # ou par type de ressource : ├── network.tf # ressources réseau ├── compute.tf # VMs, etc. ├── storage.tf # comptes de stockage ├── prod.tfvars # secrets → hors git ! ├── prod.tfvars.tpl # template → dans git ├── prod.tfbackend # config backend ├── .terraform.lock.hcl # → dans git └── .gitignore

🔐

Gestion des secrets

Ne jamais mettre de secrets en dur dans le code !

# Option 1 : Variables env (CI/CD) export ARM_CLIENT_SECRET="xxxx" export TF_VAR_db_password="xxxx" # Option 2 : Fichier .tfvars hors git db_password = "xxxx" terraform apply -var-file="secrets.tfvars" # Option 3 : Azure Key Vault (recommandé) data "azurerm_key_vault_secret" "pass" { name = "db-password" key_vault_id = azurerm_key_vault.kv.id } # Utilisation : data.azurerm_key_vault_secret.pass.value
🔒

Fixer les versions

# Toujours fixer les versions ! terraform { required_version = "~> 1.7" # >= 1.7 et < 2.0 required_providers { azurerm = { source = "hashicorp/azurerm" version = "= 3.95.0" # version exacte } } } # Opérateurs de version # = 3.95.0 → exact # ~> 3.90 → >= 3.90 et < 4.0 (patch OK) # >= 3.0 → min seulement
📌 : Commiter le .terraform.lock.hcl dans git pour garantir les mêmes versions à tous les membres de l'équipe.

1. Quelle commande doit être relancée après chaque modification du bloc terraform {} ou provider {} ?
terraform plan
terraform init
terraform validate
terraform apply
✅ terraform init télécharge les providers et configure le backend. Il doit être relancé à chaque changement du bloc terraform {}.
2. Dans le plan Terraform, quel symbole indique qu'une ressource va être détruite puis recrée (replace) ?
~
+
-/+
!
✅ -/+ = destroy + recreate. + = création, - = destruction, ~ = mise à jour in-place.
3. Pourquoi préférer for_each plutôt que count ?
for_each est plus rapide à exécuter
for_each utilise des clés stables — évite la destruction en cascade lors de modifications
count ne fonctionne pas avec Azure
for_each supporte les listes, count non
✅ Avec count, les ressources sont indexées [0], [1]… Supprimer le 2e élément sur 3 provoque la destruction/recréation du 3e. for_each utilise des clés nommées, stables.
4. Le fichier terraform.tfstate doit-il être commité dans git ?

Non — il peut contenir des secrets en clair et crée des conflits
Oui, uniquement en environnement de développement
Seulement si chiffré
✅ Le state ne doit jamais être dans git. Pour le travail en équipe, utiliser un backend distant (Azure Blob, S3…). Ajouter *.tfstate au .gitignore.
5. Quelle est la syntaxe pour référencer un attribut d'une ressource dans HCL ?

✅ Ex: azurerm_resource_group.rg.name ou azurerm_subnet.snet.id. Cette référence crée automatiquement une dépendance implicite.
6. Que fait la commande terraform state rm azurerm_resource_group.rg ?
Supprime le Resource Group dans Azure
Retire la ressource du state sans la supprimer dans Azure
Équivalent à terraform destroy -target
Supprime le fichier tfstate
✅ state rm retire la ressource uniquement du state Terraform. La ressource continue d'exister dans Azure. Terraform ne la gérera plus, sauf si on la réimporte.
7. Quel bloc HCL permet de lire une ressource existante sans la créer ni la modifier ?
resource
locals
data

✅ Le bloc data est en lecture seule. Il interroge le provider pour récupérer des attributs d'une ressource existante. Référence : data.TYPE.NOM.ATTRIBUT.
8. Quelle est la différence entre l'approche déclarative et impérative en IAC ?
Déclaratif = scripts bash ; Impératif = Terraform
Déclaratif décrit l'état souhaité (Terraform) ; Impératif décrit les étapes à exécuter (scripts)
Déclaratif est plus rapide à exécuter
Il n'y a pas de différence significative
✅ Terraform est déclaratif : on décrit ce qu'on veut (état final), et il calcule comment y arriver. L'impératif (scripts, Ansible en mode playbook) décrit les étapes précises dans l'ordre.
