🎯 C'est quoi cette MSP ?
Une MSP (Mise en Situation Professionnelle) de 3 jours où tu vas construire, entièrement avec Terraform, une infrastructure Azure complète. L'architecture s'appelle Hub & Spoke : un réseau central (Hub) avec un firewall qui distribue les connexions vers des zones séparées (Spokes).
🏛

Hub – Réseau central

Le "cœur" du réseau. Contient le firewall OPNsense qui filtre tout le trafic entrant/sortant.

🔨

Spoke DevOps Tools

Zone outils : une VM Bastion (rebond SSH) + l'appli AzureNamingTools en container Docker.

🌐

Spoke WordPress

Zone site web : une VM WordPress avec une BDD MySQL Flexible Azure.

🔧

Ce que tu vas maîtriser

Terraform, VNets, peering, NSG, routing, firewall, DNS privé/public, Docker, ACR, private endpoints, MySQL.

⚠ Limitations environnement école :
• Un seul groupe de ressources attribué (en prod il y en aurait plusieurs)
• Région autorisée : France Central uniquement
• Tag user obligatoire sur toutes les ressources
• Firewall OPNsense (pas Azure Firewall, trop cher)
• En cas de blocage "non conforme" → appelle le formateur

Ecrire 00_main.tf — Configuration du projet

Le point de départ de tout projet Terraform

▼
🎒 C'est quoi un provider ? Un provider est un plugin Terraform qui sait "parler" à un service externe. Le provider azurerm traduit ton code HCL en appels API Azure. Le provider ovh fait pareil pour OVH.
00_main.tf — A adapter avec tes valeurs
terraform { required_version = ">= 1.9.8" required_providers { azurerm = { source = "hashicorp/azurerm" version = "~> 4.12" } ovh = { source = "ovh/ovh" version = "1.1.0" } } backend "local" { path = "msp.tfstate" } } provider "azurerm" { features {} use_cli = true subscription_id = var.subscription_id } provider "ovh" { endpoint = "ovh-eu" application_key = var.ovh_app_key application_secret = var.ovh_app_secret consumer_key = var.ovh_consumer_key } # Lire le groupe de ressources EXISTANT (ne le cree pas) data "azurerm_resource_group" "msp" { name = var.resource_group_name }
💡 La data azurerm_resource_group : Tu ne crées pas le groupe de ressources, il existe déjà. La data dit à Terraform : "lis les infos sans modifier". Tu accèdes ensuite à ses attributs via data.azurerm_resource_group.msp.name, .location, .tags["user"].

Peering VNet — P1-2_peering.tf

Relier les Spokes au Hub

▼
🎒 Peering VNet : Par défaut, deux VNets Azure ne peuvent pas se "voir". Le peering crée un lien bidirectionnel. Il faut créer 2 peerings par paire : Hub→Spoke ET Spoke→Hub. Pas de peering entre les 2 Spokes.
RessourceDeVersallow_forwarded_traffic

💡 allow_forwarded_traffic = true : Indispensable pour que le trafic des Spokes passe via le firewall Hub. Sans ça, Azure bloquerait le trafic "transféré".

NSG — P1-3_nsg.tf

Pare-feu de niveau sous-réseau Azure

▼
🎒 NSG (Network Security Group) : Un pare-feu Azure de niveau réseau. Il contient des règles autoriser/bloquer sur le trafic entrant/sortant d'un sous-réseau. Pour l'instant on autorise tout (pour faciliter les tests).
2 NSG à créer (un par Spoke), associés à tous les subnets des Spokes. Règles initiales : "Allow All" entrant et sortant. Priorité 100 (entrant) et 200 (sortant).
Association NSG ↔ Subnet
resource "azurerm_subnet_network_security_group_association" "bastion" { subnet_id = azurerm_subnet.spoke_devops_tools_bastion.id network_security_group_id = azurerm_network_security_group.spoke_devops_tools.id } # Repeter pour chaque subnet des Spokes

Zone DNS privée — P1-4_dns_private.tf

Résolution de noms en interne (xxx.interne.azure)

▼
🎒 Zone DNS privée : Un serveur DNS interne à Azure. Les VMs et services dans les VNets liés peuvent résoudre des noms comme firewall.monprojet.interne.azure en IP privée — sans passer par internet.
P1-4_dns_private.tf — Zone + liens aux 3 VNets
resource "azurerm_private_dns_zone" "interne" { name = "monprojet.interne.azure" resource_group_name = data.azurerm_resource_group.msp.name tags = { user = data.azurerm_resource_group.msp.tags["user"] } } # Lier aux 3 VNets avec count (plus concis que 3 ressources) resource "azurerm_private_dns_zone_virtual_network_link" "interne_links" { count = 3 resource_group_name = data.azurerm_resource_group.msp.name name = "interne-link-${count.index + 1}" private_dns_zone_name = azurerm_private_dns_zone.interne.name virtual_network_id = element([ azurerm_virtual_network.hub.id, azurerm_virtual_network.spoke_devops_tools.id, azurerm_virtual_network.spoke_wordpress_prod.id ], count.index) registration_enabled = true # Enregistrement auto des VMs ! tags = { user = data.azurerm_resource_group.msp.tags["user"] } }

3. Routes statiques — System → Routes → Configuration
Le firewall doit savoir comment joindre les Spokes. Par défaut il ne connaît que son sous-réseau LAN (192.168.1.16/28).
DestinationPasserelleDescription
192.168.10.0/24IP passerelle LAN Azure*Spoke DevOps Tools
192.168.20.0/24IP passerelle LAN Azure*Spoke WordPress
*IP passerelle LAN Azure : C'est la 1ère IP utile du subnet Firewall-priv. Pour 192.168.1.16/28, Azure réserve les 4 premières IPs, donc la passerelle est 192.168.1.17.
4. Règle NAT sortant — Firewall → NAT → Outbound
  • Mode : Hybrid Outbound NAT
  • Ajouter une règle : Interface WAN, Source : les deux Spokes
  • Translation → Address : Interface address (= IP publique WAN)
🎒 C'est quoi le NAT ? Les VMs des Spokes ont des IPs privées. Internet ne sait pas comment répondre à 192.168.x.x. Le NAT remplace l'IP source privée par l'IP publique du firewall avant d'envoyer. La réponse revient au firewall qui fait la traduction inverse.
5. NAT entrant SSH → Bastion — Firewall → NAT → Port Forward
  • Interface : WAN | Protocol : TCP | Port : 22
  • Redirect target IP : IP privée de la VM Bastion (créée à l'étape suivante)
  • Redirect target port : 22
💡 Ceci permet de taper ssh user@IP_PUBLIQUE_FIREWALL et d'être redirigé automatiquement vers la VM Bastion qui n'a pas d'IP publique.

Table de routage Azure — P2-5_udr.tf

Forcer le trafic internet des Spokes via le firewall

▼
🎒 UDR (User Defined Route) : Par défaut Azure route le trafic vers internet directement. Pour forcer le passage par le firewall, on crée une route : "tout trafic vers 0.0.0.0/0 → passe par l'IP LAN du firewall (NVA)". Cette route est associée à tous les subnets des Spokes.
P2-5_udr.tf
resource "azurerm_route_table" "alldefaultrouting" { name = "alldefaultrouting" location = var.location resource_group_name = data.azurerm_resource_group.msp.name tags = { user = data.azurerm_resource_group.msp.tags["user"] } } resource "azurerm_route" "to_firewall" { name = "all-to-nva" resource_group_name = data.azurerm_resource_group.msp.name route_table_name = azurerm_route_table.alldefaultrouting.name address_prefix = "0.0.0.0/0" next_hop_type = "VirtualAppliance" next_hop_in_ip_address = azurerm_network_interface.opnsense_trusted.private_ip_address } # Associer a TOUS les subnets des Spokes resource "azurerm_subnet_route_table_association" "bastion" { subnet_id = azurerm_subnet.spoke_devops_tools_bastion.id route_table_id = azurerm_route_table.alldefaultrouting.id } # ... repeter pour docker_instances devops, mysql, docker_instances wordpress

VM Bastion — P2-6_vm-bastion.tf

Le rebond SSH sécurisé dans le Spoke DevOps

▼
🎒 Bastion : Les VMs internes n'ont pas d'IP publique. Le Bastion est une VM "rebond" : tu te connectes d'abord en SSH sur le Bastion, puis depuis lui tu accedes aux autres VMs en interne.
P2-6_vm-bastion.tf
resource "azurerm_network_interface" "bastion" { name = "nic-bastion" location = var.location resource_group_name = data.azurerm_resource_group.msp.name tags = { user = data.azurerm_resource_group.msp.tags["user"] } ip_configuration { name = "nic-subnet-bastion" subnet_id = azurerm_subnet.spoke_devops_tools_bastion.id private_ip_address_allocation = "Dynamic" } } resource "azurerm_linux_virtual_machine" "bastion" { name = "vm-bastion" location = var.location resource_group_name = data.azurerm_resource_group.msp.name size = "Standard_B1ls" admin_username = "tonuser" tags = { user = data.azurerm_resource_group.msp.tags["user"] } network_interface_ids = [azurerm_network_interface.bastion.id] disable_password_authentication = true admin_ssh_key { username = "tonuser" public_key = file("~/.ssh/id_rsa.pub") } os_disk { caching = "ReadWrite" storage_account_type = "Standard_LRS" disk_size_gb = 30 } source_image_reference { publisher = "Canonical" offer = "ubuntu-24_04-lts" sku = "server" version = "latest" } } resource "ovh_domain_zone_record" "bastion" { zone = var.ovh_zone subdomain = "bastion" fieldtype = "A" ttl = 3600 target = azurerm_public_ip.opnsense.ip_address }

Depuis ta machine
ssh tonuser@bastion.tondomaine.ovh # Le firewall redirige le port 22 vers le Bastion # Transferer ta cle privee sur le Bastion (pour rebondir vers d'autres VMs) scp ~/.ssh/id_rsa tonuser@bastion.tondomaine.ovh:~/.ssh/id_rsa ssh tonuser@bastion.tondomaine.ovh "chmod 600 ~/.ssh/id_rsa"

Importer les ressources ARM dans Terraform — P2-2_import.tf

Récupérer la gestion Terraform des ressources créées par ARM

▼
🎒 Import Terraform : Terraform ne "connaît" que les ressources qu'il a créées. L'import dit : "cette ressource HCL = cette ressource Azure existante".
⚠ Ordre d'import : 1. IP publique → 2. NSG → 3. Interfaces réseau → 4. Associations NSG↔NIC → 5. VM
Trouver l'ID Azure d'une ressource
Portail Azure → ressource → Vue JSON → copie le champ "id" tout en haut.
Exemple — import IP publique
resource "azurerm_public_ip" "opnsense" { name = "OPNsense-PublicIP" location = var.location resource_group_name = data.azurerm_resource_group.msp.name allocation_method = "Static" tags = { user = data.azurerm_resource_group.msp.tags["user"] } } import { to = azurerm_public_ip.opnsense id = "/subscriptions/XXXX/resourceGroups/TON_RG/providers/Microsoft.Network/publicIPAddresses/OPNsense-PublicIP" }
Processus pour chaque ressource
